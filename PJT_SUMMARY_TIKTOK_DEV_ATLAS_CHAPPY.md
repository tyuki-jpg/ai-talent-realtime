# PJTサマリ: AIタレント TikTok配信対応（Atlas チャッピー連携）

作成日: 2026-02-24

## 1. PJT目的
AIタレント配信基盤を TikTok でも運用可能にするため、TikTok Developer への登録と連携要件を整理し、Atlas のチャッピーへ実装判断可能な形で連携する。

## 2. 背景
現行プロトタイプは `LiveAvatar + OpenAI` をベースに、ローカルでリアルタイム会話配信を確認済み。  
次フェーズとして、配信チャネルを TikTok に拡張する。

## 3. 現行システム要約
1. フロント: 音声入力/手入力、モード切替（FULL/CUSTOM）
2. サーバー: LiveAvatar セッション作成・keepalive・stop・speak、OpenAI応答生成
3. 主なエンドポイント:
`/liveavatar/new-session`, `/liveavatar/keepalive`, `/liveavatar/stop`, `/liveavatar/speak`, `/reply`, `/persona`

## 4. 今回の対象スコープ
1. TikTok Developer アカウント/アプリ登録
2. 配信連携方式の決定
3. Atlas チャッピー向けに、実装インプット（要件・接続情報・運用方針）を提供

## 5. TikTok連携で確定すべき事項（チャッピーへ連携）
1. 連携方式:
TikTok LIVE API/SDKで直接連携するか、RTMP中継経由で連携するか
2. 認証方式:
OAuth利用有無、トークン更新手順、権限スコープ
3. 配信フロー:
配信開始/停止トリガー、障害時の再接続、セッション有効期限管理
4. コメント連携:
TikTokコメント取得、LLM入力フィルタ、NGワード運用
5. 音声方針:
LiveAvatar Voice（FULL）を標準とし、CUSTOMは代替経路として扱う
6. 運用監視:
配信状態監視、失敗時アラート、ログ保存期間

## 6. 推奨実行ステップ
1. TikTok Developer登録を完了し、利用可能API/権限を確定
2. Atlas チャッピーへ「連携方式の候補比較（LIVE API vs RTMP）」を提示し方式決定
3. 方式確定後、最小PoCを実装
4. コメント受信→AI応答→発話→配信反映までのE2E検証
5. 運用設計（トークン更新、障害時復旧、監視）を確定

## 7. リスクと対策
1. API利用制限/審査遅延:
先にRTMP代替案を準備し、PoC停止リスクを回避
2. 認証期限切れ:
期限監視と自動更新ジョブを設計
3. 配信遅延/音ズレ:
音声経路を固定し、配信遅延を計測してしきい値管理
4. モデレーション不備:
入力フィルタと運用ルールを先行導入

## 8. 成果物（チャッピー連携物）
1. TikTok Developer 登録情報（アプリID、権限、審査状態）
2. 連携方式決定メモ（採用理由、非採用理由）
3. API連携仕様メモ（認証、イベント、エラー処理）
4. PoC検証結果（成功条件、未解決課題、次アクション）

## 9. 直近アクション
1. TikTok Developer 登録完了情報を取得
2. チャッピーへ本サマリと方式選定論点を共有
3. 方式決定後、実装タスクをチケット化して着手
